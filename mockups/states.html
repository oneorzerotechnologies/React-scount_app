Notes

One empty-state component, six surfaces. The icon-on-tinted-square + headline + body + primary CTA + escape hatch pattern in 09A is exactly the same shape we use on empty Invoices, empty Contacts (per segment), empty Notifications ("All caught up" with a moss check), and empty filtered lists. The team builds <EmptyState /> once and parameterises icon + copy + CTA.

Offline policy on v1. Reads stay available from the TanStack Query cache; mutations are blocked at the UI level (disabled CTAs + a one-line caption). A proper offline-mutation queue with conflict resolution is Phase 2 — for now, the user knows what's cached and that they can't write.

Forced update is the only screen without a tab bar. Everything else keeps it for orientation. The forced-update screen is reached when GET /v1/version-check returns force_update; we never let the user navigate past it. The "support@scount.my" line is the escape hatch for users on a platform where the store is unavailable (jailbroken, sideloaded, etc.).

Errors carry a trace ID. Sentry's event ID surfaces inline as a dim mono line (trace · b9c4e2f). Users can quote it in support tickets; we look it up and have the full trace + breadcrumb trail without asking the user a single follow-up question.

409s use the bottom sheet pattern. Not a toast (too short), not an alert (too modal). The sheet quotes the API's message verbatim and routes the user to wherever they can resolve it. Same component for "can't delete a converted quote", "can't edit an accepted quote", "can't delete a paid invoice".

Permissions: prevent first, react second. The mobile UI hides what the user can't do — the pencil Edit button, Delete in the overflow menu, the "+ New" CTA in lists — based on a permissions object the backend ships in workspace.settings. A read-only viewer (typically MEMBER role) sees a small "Read-only · Member role" pill near the status badge and an info banner that names the role + what's required. If the user still hits a 403 (race condition, role just changed, edge case), the same bottom-sheet pattern as 09F appears — copy names the role and the action, primary CTA is "Message owner" with a pre-filled email/note. The client never reimplements role → permission logic; it just respects the boolean map the backend computes.

Skeleton, not spinner. Skeletons preview the shape of the result, so the perceived latency drops even at the same wall-clock time. Use them everywhere the result has a known shape (lists, detail pages, dashboard cards). Spinners are reserved for "I genuinely don't know how long this'll take" — basically just biometric prompts and PDF generation.
